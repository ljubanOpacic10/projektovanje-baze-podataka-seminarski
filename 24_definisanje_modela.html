Definisanje modela – od klase do tabele

U prethodnoj lekciji upoznali smo Entity Framework Core i naučili šta su DbContext i DbSet. Sada je vreme da detaljnije pogledamo kako se definišu model klase, odnosno kako od klase u C# nastaje tabela u bazi podataka.

Klasa kao opis tabele

U Code-First pristupu, svaka klasa predstavlja jednu tabelu. Svojstva klase postaju kolone u tabeli. Ovo smo već videli na jednostavnom primeru, ali sada ćemo dodati detalje koji su važni za pravilan rad sa bazom. Počnimo od klase Ucenik: public class Ucenik { public int Id { get; set; } public string Ime { get; set; } public string Prezime { get; set; } public string Razred { get; set; } } EF Core prepoznaje ovu klasu i na osnovu nje kreira tabelu. Svako svojstvo postaje kolona:
Svojstvo u klasi Kolona u tabeli Tip podatka
Id Id INT
Ime Ime NVARCHAR
Prezime Prezime NVARCHAR
Razred Razred NVARCHAR

Primarni ključ u EF Core-u

Setimo se: svaka tabela mora da ima primarni ključ. U SQL-u smo to označavali sa PRIMARY KEY. Kako EF Core zna koje svojstvo je primarni ključ? EF Core primenjuje konvenciju: ako klasa ima svojstvo koje se zove Id ili [ImeKlase]Id, ono automatski postaje primarni ključ. Na primer: // EF Core prepoznaje "Id" kao primarni ključ public int Id { get; set; } // Isto važi i za "UcenikId" public int UcenikId { get; set; } Pored toga, EF Core automatski podešava da se vrednost primarnog ključa generiše sama, slično kao IDENTITY u SQL Server-u. To znači da ne moramo ručno da zadajemo Id kada dodajemo novog učenika, baza sama dodeljuje sledeći broj. Uporedi sa SQL-om koji smo ranije pisali: -- u SQL-u smo ručno davali Id INSERT INTO Ucenik (id_ucenik, ime, prezime, razred) VALUES (1, 'Marko', 'Markovic', 'I1'); U EF Core-u to nije potrebno: // Id se automatski generiše var ucenik = new Ucenik { Ime = "Marko", Prezime = "Markovic", Razred = "I1" };

Veze između klasa – navigaciona svojstva

U ranijim lekcijama učili smo o relacijama između entiteta. Na primer, učenik ima više ocena, a svaka ocena pripada jednom učeniku. U SQL-u smo to rešavali pomoću stranog ključa. Kako to izgleda u EF Core-u? Pogledajmo klasu Ocena: public class Ocena { public int Id { get; set; } public int Vrednost { get; set; } public DateTime Datum { get; set; } // strani ključ public int UcenikId { get; set; } public int PredmetId { get; set; } // navigaciono svojstvo public Ucenik Ucenik { get; set; } public Predmet Predmet { get; set; } } Svojstvo UcenikId je strani ključ. Ono čuva Id učenika kome ocena pripada, isto kao što smo u SQL-u imali kolonu id_ucenik u tabeli Ocena. Svojstvo Ucenik (tipa Ucenik) je navigaciono svojstvo. Ono omogućava da iz objekta Ocena direktno pristupimo objektu Ucenik kome ta ocena pripada.

Navigaciona svojstva sa druge strane

Klasa Ucenik može da ima listu svojih ocena: public class Ucenik { public int Id { get; set; } public string Ime { get; set; } public string Prezime { get; set; } public string Razred { get; set; } // jedan učenik ima više ocena public List<Ocena> Ocene { get; set; } } Svojstvo Ocene tipa List govori EF Core-u da jedan učenik može imati više ocena. To je relacija 1:N koju smo crtali u ER dijagramu. EF Core na osnovu ovih klasa automatski kreira tabele sa odgovarajućim stranim ključem: -- EF Core generiše nešto poput ovoga: CREATE TABLE Ucenici ( Id INT PRIMARY KEY IDENTITY, Ime NVARCHAR(MAX), Prezime NVARCHAR(MAX), Razred NVARCHAR(MAX) ); CREATE TABLE Ocene ( Id INT PRIMARY KEY IDENTITY, Vrednost INT, Datum DATETIME2, UcenikId INT FOREIGN KEY REFERENCES Ucenici(Id) ); Primeti kako je to veoma slično onome što smo mi ručno pisali u SQL-u. Razlika je samo u tome što sada ne moramo to da radimo sami.

Kompletan model školskog sistema

Hajde da pogledamo kako bi izgledao kompletan model za naš školski sistem sa četiri klase: public class Ucenik { public int Id { get; set; } public string Ime { get; set; } public string Prezime { get; set; } public string Razred { get; set; } public List<Ocena> Ocene { get; set; } } public class Profesor { public int Id { get; set; } public string Ime { get; set; } public string Prezime { get; set; } public List<Predmet> Predmeti { get; set; } } public class Predmet { public int Id { get; set; } public string Naziv { get; set; } public int ProfesorId { get; set; } public Profesor Profesor { get; set; } public List<Ocena> Ocene { get; set; } } public class Ocena { public int Id { get; set; } public int Vrednost { get; set; } public DateTime Datum { get; set; } public int UcenikId { get; set; } public Ucenik Ucenik { get; set; } public int PredmetId { get; set; } public Predmet Predmet { get; set; } } Pogledaj kako se ovde jasno vide sve relacije. Profesor predaje više predmeta. Predmet pripada jednom profesoru. Učenik ima više ocena. Svaka ocena je vezana za jednog učenika i jedan predmet. To su iste relacije koje smo definisali u ER dijagramu, samo izražene kroz kod.
Česta greška kod učenika je da zaborave da dodaju navigaciono svojstvo ili strani ključ. Ako napišeš samo UcenikId bez svojstva Ucenik, EF Core će i dalje napraviti vezu, ali nećeš moći da u kodu jednostavno pristupiš učeniku iz objekta ocene. Najbolja praksa je da uvek imaš oba: strani ključ (int UcenikId) i navigaciono svojstvo (Ucenik Ucenik).
