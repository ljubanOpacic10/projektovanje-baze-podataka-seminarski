Migracije – kreiranje baze iz koda

U prethodnoj lekciji naučili smo kako da definišemo model klase u C# koje predstavljaju tabele u bazi. Napisali smo klase Ucenik, Profesor, Predmet i Ocena sa svim svojstvima i vezama. Ali te klase za sada postoje samo u kodu. Kako od tog koda zaista nastaje baza podataka sa tabelama? Odgovor su migracije.

Šta su migracije

Migracija je način da kažemo EF Core-u: „Pogledaj moje klase i na osnovu njih napravi ili ažuriraj bazu podataka." Migracija je u suštini skup instrukcija koje opisuju koje promene treba primeniti na bazu. Te instrukcije generiše sam EF Core na osnovu naših klasa. Mi ne moramo da pišemo SQL komande za kreiranje tabela, ali možemo da vidimo šta EF Core generiše u pozadini. Možemo da zamislimo migracije kao „istoriju promena" naše baze. Svaka migracija beleži šta je promenjeno: da li je dodata nova tabela, novo svojstvo ili nova veza.

Tok rada sa migracijama

Pogledajmo kako izgleda ceo proces od pisanja klase do kreiranja baze:
Korak Šta radiš Rezultat
1 Napišeš klase (modele) Ucenik, Profesor, Predmet, Ocena
2 Napišeš DbContext SkolaContext sa DbSet svojstvima
3 Pokreneš komandu za kreiranje migracije EF Core generiše fajl sa instrukcijama
4 Pokreneš komandu za primenu migracije Baza se kreira ili ažurira

Komande za migracije

Za rad sa migracijama koriste se dve osnovne komande u konzoli: dotnet ef migrations add KreiranjeBaze Ova komanda govori EF Core-u da pogleda naše klase i generiše migraciju. Ime migracije „KreiranjeBaze" opisuje šta ta migracija radi. Možeš da izabereš bilo koje ime koje jasno opisuje promenu. dotnet ef database update Ova komanda primenjuje migraciju, odnosno izvršava generisane SQL komande nad bazom. Ako baza ne postoji, kreiraće je. Ako postoji, primeniti samo nove promene.

Šta migracija generiše u pozadini

Kada pokrenemo prvu migraciju za naš školski sistem, EF Core generiše kod koji opisuje kreiranje tabela. Fajl migracije izgleda otprilike ovako: public partial class KreiranjeBaze : Migration { protected override void Up(MigrationBuilder migrationBuilder) { migrationBuilder.CreateTable( name: "Ucenici", columns: table => new { Id = table.Column<int>(nullable: false) .Annotation("SqlServer:Identity", "1, 1"), Ime = table.Column<string>(nullable: true), Prezime = table.Column<string>(nullable: true), Razred = table.Column<string>(nullable: true) }, constraints: table => { table.PrimaryKey("PK_Ucenici", x => x.Id); }); } } Ne moraš da razumeš svaki detalj ovog koda. Ono što je bitno je da vidiš da EF Core sam opisuje kreiranje tabele sa kolonama, tipovima podataka i primarnim ključem. U pozadini, ovo se prevodi u SQL: CREATE TABLE Ucenici ( Id INT IDENTITY(1,1) PRIMARY KEY, Ime NVARCHAR(MAX), Prezime NVARCHAR(MAX), Razred NVARCHAR(MAX) ); Prepoznaješ li ovo? To je gotovo ista komanda koju smo ručno pisali u ranijim lekcijama. Razlika je samo u tome što sada EF Core to radi umesto nas.

Poređenje sa ručnim CREATE TABLE

Hajde da direktno uporedimo dva pristupa:
Database-First (SQL) Code-First (migracija)
Pišeš CREATE TABLE ručno EF Core generiše SQL iz klase
Moraš da znaš tačnu SQL sintaksu Definišeš svojstva u C# klasi
Promene radiš sa ALTER TABLE Dodaš novo svojstvo i napraviš novu migraciju
Izvršavaš u SSMS-u Izvršavaš komandom u konzoli

Dodavanje promena – nova migracija

Zamisli da nakon kreiranja baze shvatiš da u klasi Ucenik treba dodati i email adresu. U Database-First pristupu bi morao da napišeš: ALTER TABLE Ucenik ADD email NVARCHAR(100); U Code-First pristupu, samo dodaš novo svojstvo u klasu: public class Ucenik { public int Id { get; set; } public string Ime { get; set; } public string Prezime { get; set; } public string Razred { get; set; } public string Email { get; set; } // novo svojstvo } I zatim pokreneš novu migraciju: dotnet ef migrations add DodavanjeEmaila EF Core sam prepozna da je dodato novo svojstvo i generiše migraciju koja dodaje kolonu u postojeću tabelu. Kada primenimo migraciju, baza se automatski ažurira.

Zaključak

Ovo je možda najvažnija stvar u celoj lekciji. Ceo proces izgleda ovako: Napišeš klase u C#. Pokreneš jednu komandu za generisanje migracije. Pokreneš drugu komandu za primenu migracije. Baza sa svim tabelama, kolonama, ključevima i relacijama sada postoji. Nisi napisao nijednu liniju SQL-a. Nisi otvarao SSMS. Sve se desilo iz koda. Naravno, u pozadini SQL i dalje postoji. EF Core ga generiše i izvršava umesto tebe. Ali moć ovog pristupa je u tome što sve promene pratiš na jednom mestu, u kodu, a baza se automatski usklađuje.
Česta zabuna je da učenici misle da migracije brišu postojeće podatke. To uglavnom nije slučaj. Migracije samo dodaju ili menjaju strukturu baze. Postojeći podaci ostaju. Međutim, ako obrišeš kolonu koja sadrži podatke, ti podaci će biti izgubljeni. Zato je važno pažljivo razmisliti pre nego što napraviš migraciju koja uklanja nešto iz modela.

Pregled komandi

Za kraj, evo pregled najvažnijih komandi za rad sa migracijama:
Komanda Šta radi
dotnet ef migrations add [Ime] Kreira novu migraciju
dotnet ef database update Primenjuje migracije na bazu
dotnet ef migrations list Prikazuje listu svih migracija
dotnet ef migrations remove Uklanja poslednju migraciju

Šta sledi dalje

U narednoj lekciji ćemo videti kako da radimo sa podacima kroz EF Core. Naučićemo kako da dodajemo, čitamo, menjamo i brišemo podatke koristeći C# kod umesto SQL komandi.
Objasni svojim rečima šta je migracija i čemu služi. Napiši redosled komandi koje bi koristio da od klasa Ucenik i Profesor kreiraš bazu podataka koristeći EF Core. Zamisli da trebaš da dodaš kolonu BrojTelefona u tabelu Profesor. Opiši korake koje bi pratio u Code-First pristupu. Uporedi proces kreiranja tabele ručno (SQL) i automatski (migracija).
